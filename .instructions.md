# AI'll Be Sick Copilot Instructions (Operational Overlay)

This file is the concise operating layer for day-to-day Copilot requests.
Use this together with `AGENTS.md` files, where `AGENTS.md` provides deeper reference details.

## Canonical architecture facts

- Monorepo with `backend/` (Flask) and `frontend/` (Next.js App Router).
- Backend local runtime: `http://localhost:10000` using `python run.py`.
- Frontend local runtime: `http://localhost:3000`.
- Backend URL handling in frontend is centralized in `frontend/utils/backend-url.ts` (includes localhost normalization behavior for server-side execution).

## Frontend implementation defaults

- Use TypeScript for all new frontend code.
- Use Server Components for data fetching when possible.
- Use Server Actions (`next-safe-action`) for mutations.
- Pair every mutation action with a Zod schema in `frontend/schemas/`.
- Use `actionClient.inputSchema(...).action(...)` from `frontend/actions/client.ts`.
- After mutation, apply cache revalidation with `revalidatePath` and/or `revalidateTag` as appropriate.
- Return structured success/error objects; avoid uncaught throws for expected failures.

## Backend implementation defaults

- Keep routes in Flask blueprints under `backend/app/api/`.
- Keep reusable logic in `backend/app/services/`.
- Read tunables/thresholds from `backend/app/config.py` and environment variables (avoid hardcoding threshold literals in endpoints).
- Keep endpoint failures explicit and safe for API consumers.

## UI and copy defaults

- Prefer existing reusable components and established DaisyUI patterns before introducing new UI patterns.
- Build UI changes mobile-first, then scale up to larger breakpoints.
- Keep tap targets and interactive controls usable on small screens.
- Use `lucide-react` for icons.
- For user-facing medical text: use plain language, calm tone, and avoid certainty claims.
- Avoid using the word "cluster" in user-facing text; use "group".

## Fool-Proof + Clinical Rigor Framework

### Readability Targets

**Patient/General UI (non-medical explanations):**

- Target reading level: Grade 6–8 (Flesch–Kincaid index ~60–70).
- Short sentences (~8–18 words), one action per sentence.

**Urgent warnings & safety banners:**

- Target reading level: Grade ≤6 (Flesch–Kincaid index ~75+).
- One clear directive per message; no nested conditions.

**Clinician-facing panels & advanced features:**

- Target reading level: Grade 10–12.
- Include clinical jargon but pair with help tooltips or inline glossary links.

**Validation:** Before shipping any user-facing copy, use plain-language review and avoid stacking medical terms. If you cannot explain a term in 1 sentence for a Grade-6 reader, add a help link or tooltip.

### Error Recovery Safeguards (REQUIRED)

**Input Preservation**

- Never clear form fields, chat history, or selected answers when an error occurs.
- Show the error message above/below the input, allow correction, and retry.

**Error Messages & Actions**

- Show plain-language error + ONE clear next action:
  - `Try again` (for transient failures: network timeout, rate limit)
  - `Go back` (for navigation-level errors)
  - `Use manual option` (for unavailable features)
  - `Contact support` (for unrecoverable errors with code)
- Never show raw error codes or stack traces to patients.

**Duplicate Submission Prevention**

- Disable submit button while pending; show invisible counter to prevent double-clicks.
- Implement server-side idempotency (e.g., unique request IDs) for diagnosis and chat submissions.

**Transient Failure Retry**

- For network/timeout failures, offer up to 3 retries with visible countdown or progress indicator.
- Back off exponentially (1s, 2s, 4s) between retries.

**Fallback & Offline States**

- When backend unreachable: show "We're having trouble connecting. Your answers are saved locally" + option to retry.
- Do NOT prevent form input; let users continue and sync when available.

**Risky Action Confirmations**

- Require explicit confirmation for irreversible actions (e.g., "Clear all my results?", "Delete this case?").
- Show confirmation as a modal with two buttons: `Cancel` (primary/safe), `Yes, proceed` (secondary, clear action).
- Provide undo for destructive edits where technically feasible (e.g., restore previous chat state).

**Field-Level Validation**

- Validate and report specific field errors immediately (on blur or after 500ms typing delay).
- Show error inline next to the field, not in a separate validation summary.
- Example: "Symptom list cannot be empty. Add at least one symptom." (not "Invalid input form.")

### Novice Usability Validation Workflow (Gate Before Merge)

**Applies to:**

- Patient-facing diagnosis flows (new or substantially changed).
- Critical warnings, safety messages, or follow-up question rewrites.
- Any UI that non-technical users (age 18–70, minimal health literacy) interact with.

**Validation Process:**

1. Define 3–5 critical user tasks (e.g., "Start a symptom assessment," "Understand your results," "Know when to seek care," "Find next action").
2. Test with 5–8 unpaid volunteer participants who:
   - Have never used the app.
   - Receive zero coaching or guidance.
   - Span age 20–70 and mixed tech comfort.
3. Measure:
   - **Task completion rate:** Did users complete the task without giving up? (Target: ≥90%)
   - **Critical mistakes:** Did users misunderstand a safety warning or take a wrong action? (Target: 0 per task)
   - **Time to complete:** Baseline for future improvements (target: <2 min per critical task).
   - **Comprehension:** Ask "What should you do next?" after results—do they understand without rewording? (Target: ≥90% correct answer)

**Pass/Fail Gates:**

- ✅ **PASS:** Task completion ≥90%, zero safety-critical misunderstandings, all users understand next steps.
- ❌ **BLOCK MERGE:** Task completion <90%, any safety misunderstandings (e.g., "I think I'm definitely sick" when result said "might indicate"), or users unable to find next action.
- 📝 **If failed:** Document findings in PR, fix affected copy/UX, and retest with same cohort before re-merging.

**Exemptions:**

- Internal tools, clinician-only panels, or developer-facing documentation.
- Minor tweaks to existing text (≤10 words changed) if no logic changes.
- Backend-only changes.

**Documentation in PR:**

- Link novice test recording/notes if available (anonymized).
- Note any edge cases discovered and how they were addressed.

## Validation defaults

- Frontend quick validation: `npx tsc --noEmit` in `frontend/`.
- Backend tests: run targeted `pytest` files under `backend/tests/` relevant to the change.

## Automated CI/CD Workflows

GitHub Actions workflows automatically validate all PRs and pushes. If you update these workflows, keep them in sync with `.instructions.md` validation defaults:

- **Frontend checks** (`.github/workflows/frontend-checks.yml`):
  - TypeScript type check (`npx tsc --noEmit`)
  - Next.js linting (`npm run lint`)
  - Production build verification (`npm run build`)
  - **Updated if:** TypeScript/Next.js commands change or new linting tools are added

- **Backend checks** (`.github/workflows/backend-checks.yml`):
  - pytest test suite (`python -m pytest tests/`)
  - Python linting (flake8)
  - Flask app initialization check
  - **Updated if:** pytest structure changes, new linting tools added, or Flask bootstrap logic changes

**Important:** When these workflows change, update `.instructions.md` validation defaults as the source of truth for what developers should run locally.
