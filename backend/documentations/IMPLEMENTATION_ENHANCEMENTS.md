# Implementation Enhancements & Supplementary Methodology

This document details the advanced technical features and architectural enhancements implemented in the final codebase of the **AI'll Be Sick** diagnostic system that extend beyond the core methodologies documented in the original thesis manuscript.

While the original thesis accurately describes the foundational inference pipeline, multi-language transformer application, and Monte Carlo Dropout (MCD) for epistemic uncertainty, the engineering requirements of deploying a robust, clinical-grade medical application necessitated several advanced additions. These additions do not alter the foundational research; rather, they mathematically operationalize the concepts (such as the "adaptive questionnaire") and ensure higher bounds of clinical safety.

---

## 1. Neuro-Symbolic Verification (Phenotype Clamping)

### Description
The system incorporates a "Neuro-Symbolic" verification layer that cross-references the purely neural output (from BioClinical ModernBERT or RoBERTa Tagalog) against structured, symbolic medical knowledge. 

### How it Functions
When the AI model outputs probability distributions across the 6 supported diseases, the system intercepts these probabilities. It parses the user's input text to identify specific symptom concepts. It then verifies if the user's presented symptoms belong to the recognized phenotype (symptom profile) of the predicted disease. 
If the model predicts a disease, but the user presents symptoms that are medically impossible or highly atypical for that disease (unexplained concepts), the system performs a "clamp." The probability of that impossible disease is mathematically forced to near-zero (`1e-10`), and the Softmax distribution is recalculated.

### System Integration
This occurs in `backend/app/api/diagnosis.py` via the `_clamp_impossible_diseases` function. It acts as a hard safety mechanism, preventing the AI from hallucinating a diagnosis that contradicts basic medical logic, and it stops the follow-up questionnaire from wasting the user's time on an impossible track.

## 2. Adaptive Questionnaire Engine: Bayesian Updates & EIG

### Description
The thesis manuscript proposed an "adaptive questionnaire" to handle cases where initial prediction confidence is low. The codebase implements the actual logic for this adaptivity using probability theory, specifically **Bayesian Inference** and **Expected Information Gain (EIG)**.

### How it Functions
*   **Bayesian Evidence Update:** Each question in the system's `question_bank.json` possesses a pre-calculated statistical "weight" associating it with a specific disease. When a user answers a follow-up question ("Yes" or "No"), this response is treated as new evidence ($E$). The system mathematically updates the prior probability distribution ($P(H)$) generated by the AI into a posterior probability ($P(H|E)$) using Bayes' Theorem.
*   **Expected Information Gain (EIG):** To decide *which* question to ask next, the system simulates the utility of every available question. It selects the question that is mathematically expected to yield the highest reduction in **Predictive Entropy** across the disease distribution. It asks the question most likely to push the system toward a definitive conclusion.

### System Integration
Integrated deeply into the follow-up dialogue state manager (`backend/app/api/diagnosis.py` and `backend/app/services/information_gain.py`). This structural enhancement ensures that the dialogue is not a static tree but a dynamic, mathematically optimized path to the shortest possible confident diagnosis.

## 3. Comprehensive Uncertainty Suite (Multi-Metric)

### Description
The thesis relies heavily on Mutual Information (MI) as its primary epistemic uncertainty metric. In production, relying on a single metric can be brittle. The implementation expands this to a comprehensive 6-metric suite.

### How it Functions
Using the exact same 50 iterations of Monte Carlo Dropout outputs proposed in the thesis, the system calculates:
1.  **Mutual Information:** (As promised) Model doubt.
2.  **Predictive Entropy:** Total uncertainty.
3.  **Variance:** Prediction stability across the 50 stochastic runs.
4.  **Coefficient of Variation:** Relative uncertainty normalized by the probability scale.
5.  **Ensemble Disagreement:** The rate at which the 50 stochastic forward passes predict fundamentally different disease classes.
6.  **Max Probability Standard Deviation:** Identifies unstable decision boundaries.

### System Integration
Implemented in `backend/app/services/ml_service.py` via the `compute_multi_metric_uncertainty` method. The system can be configured to flag a prediction for human clinical review if *any* of the critical metrics (MI, Variance, Disagreement) exceed validated thresholds, drastically improving edge-case detection.

## 4. Continuous Calibration & Temperature Scaling

### Description
Neural networks, including Transformers, are notoriously overconfident (e.g., they might output 95% confidence but only be right 70% of the time). To ensure the reported confidence scores are clinically trustworthy, the implementation includes calibration mechanisms.

### How it Functions
*   **Expected Calibration Error (ECE):** The system continuously measures the gap between predicted probability and actual empirical accuracy.
*   **Temperature Scaling:** The system calculates an optimal temperature variable ($T$). By dividing the raw model logits by $T$ prior to the Softmax function, it "softens" the probability distribution, effectively aligning the model's confidence with its true accuracy rate without changing the chosen prediction.

### System Integration
Located in `ml_service.py`. While the thesis elected not to use Temperature Scaling as a standalone *alternative* to MCD, the codebase utilizes it *in tandem* with MCD to ensure the baseline probability distributions feeding into the Bayesian update engine are strictly calibrated.

## 5. Dynamic Threshold Optimization & Risk Stratification

### Description
The initial academic research utilized static, empirical thresholds (e.g., Confidence < 70%, MI > 0.05) to flag unreliable predictions. The final system replaces this with data-driven optimization.

### How it Functions
*   **ROC & Precision-Recall Optimization:** The system includes utilities to automatically define optimal uncertainty cutoffs by evaluating the Receiver Operating Characteristic (ROC) and Precision-Recall curves on validation datasets. It finds the threshold that perfectly balances catching errors versus false-alarming.
*   **Clinical Risk Stratification Matrix:** Based on these optimized metrics, predictions are routed through a multi-tiered workflow:
    *   **Automated Diagnosis:** (High Conf, Low MI, Low Disagreement)
    *   **Nurse Review:** (Moderate Conf, Borderline MI)
    *   **Physician Review:** (Low Conf, High MI, High Disagreement)

### System Integration
Exposed through configuration files and evaluation scripts (`scripts/evaluate_uncertainty.py`). This evolves the prototype from a diagnostic calculator into a workflow-aware clinical support pipeline.

## 6. Dynamic CPU Quantization

### Description
A software engineering necessity for resource-constrained deployment environments.

### How it Functions
The system uses PyTorch's `torch.quantization.quantize_dynamic` targetting specifically `torch.nn.Linear` layers. This converts high-precision 32-bit floating-point weights into 8-bit integers (`qint8`). 

### System Integration
Triggered during model initialization in `ml_service.py` if no CUDA (GPU) device is found. This drasticaly reduces RAM usage and dramatically speeds up CPU inference times, ensuring the complex MCD loops execute rapidly in standard web-hosting environments, a practical necessity outside of the GPU-heavy training environments discussed in the academic manuscript.
