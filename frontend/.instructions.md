# Frontend Copilot Overlay

Use this file with root `.instructions.md`.
This file narrows guidance specifically for `frontend/` implementation work.

## Scope

- Next.js App Router frontend in TypeScript.
- Server actions, Zod schemas, Prisma reads/writes, Supabase auth, and clinician/patient UI flows.

## Implementation defaults

- Prefer Server Components for initial data fetching.
- Use Server Actions for mutations via `next-safe-action`.
- Pair mutation inputs with Zod schemas in `frontend/schemas/`.
- Keep backend URL usage centralized through `frontend/utils/backend-url.ts`.
- Keep mutation outputs structured (`success`/`error`) for resilient UX flows.

## Canonical references

- Action client: `frontend/actions/client.ts`
- Diagnosis flow: `frontend/actions/run-diagnosis.ts`
- Schema examples: `frontend/schemas/`
- Backend URL resolver: `frontend/utils/backend-url.ts`

## UI and copy

- Reuse existing components and current DaisyUI-oriented patterns.
- Apply mobile-first layouts by default, then enhance for tablet/desktop breakpoints.
- Prefer responsive spacing/typography and avoid desktop-only assumptions in interaction flows.
- Use `lucide-react` icons.
- For user-facing medical text: avoid certainty claims; use plain language and clear next steps.
- Avoid the word "cluster" in user-facing text; use "group".

## Fool-Proof + Clinical Rigor (Frontend-Specific)

### Error Recovery Implementation Patterns

**Forms & Input:**

- Use `<input>` with `value` (controlled) so user input persists on error.
- On submission error, show error banner above the form (not replacing it).
- Use `<FormError>` component for field-level errors inline next to inputs (not in a summary box).
- Disable submit button while `isPending`; show visible loading state (spinner + "Submitting...").

**Async Actions (via `next-safe-action`):**

- Always show `isError` state; map to plain-language message in UI (never raw error.message).
- Provide `retryable` flag in action response for transient vs. permanent failures.
- Implement retry button for transient failures (e.g., network timeout).
- Log permanent errors to console with safe surface messaging: "We encountered an issue. Our team has been notified."

**Diagnosis & Chat Flows:**

- Preserve chat history on error; show retry option within the conversation.
- Use optimistic updates for diagnosis submissions, but revert on error without losing user answers.
- Add confirmation modal before clearing diagnosis or chat: "Delete all your answers? You can't undo this." with `Cancel` (primary) and `Yes, delete` (secondary).

### Readability & Medical Copy

- Use clinical-copywriting skill for all symptom explanations, uncertainty messaging, and result interpretation.
- Target Grade 6–8 for all patient-facing text (avoid stacking medical terms).
- Test comprehension: can a non-technical adult explain the next step after reading your message?

### Validation Checklist

- Minimum check for frontend code changes: `npx tsc --noEmit`
- Optional broader check: `npm run build`
- **Before patient-facing UX ships:** Confirm error states tested (form with validation, network failure, timeout).
- **Diagnosis/results flows:** Confirm novice user can understand the result and knows their next step.
- **CI/CD:** Automatically runs on every PR via `.github/workflows/frontend-checks.yml` (TypeScript, linting, build check).
